Database Design Principles

Normalization Rules
  • 1NF: Eliminate repeating groups, each cell contains atomic values
  • 2NF: Remove partial dependencies, every non-key attribute depends on the entire primary key
  • 3NF: Remove transitive dependencies, non-key attributes don't depend on other non-key attributes
  • BCNF: Every determinant is a candidate key (advanced normalization)
Relationship Types
  • One-to-One: Each record in A relates to at most one record in B
  • One-to-Many: Each record in A can relate to multiple records in B
  • Many-to-Many: Multiple records in A relate to multiple records in B (requires junction table)
  • Self-Referential: A table relates to itself (hierarchies, categories)

Implementation Guidance

Knex.js Migration Strategy
Migration Files Structure:
  • Up migrations: Create/modify tables and columns
  • Down migrations: Reverse the changes (for rollbacks)
  • Naming convention: YYYYMMDD_HHMMSS_description.js
  • Version control: Never modify existing migrations
Example Migration:
exports.up = (knex) => knex.schema.createTable('users', (table) => { table.increments('id').primary(); table.string('email').unique().notNullable(); table.string('password_hash').notNullable(); table.enum('role', ['student', 'teacher', 'admin']); });
Seed Data Strategy
  • Development data: Sample users, courses, and assignments
  • Environment-specific: Different seeds for dev/staging/production
  • Relationships: Maintain referential integrity in seed data
  • Updates: Seed files should be idempotent (safe to run multiple times)
Best Practices:
  • Use faker.js for realistic test data
  • Create admin users for testing
  • Include various grade scenarios
  • Document seed data purpose

Common Database Patterns

Grade Calculation Pattern

Store raw scores and calculate final grades dynamically

  • Raw scores: Store individual assignment scores
  • Weight categories: Define category weights (Homework: 30%, Exams: 70%)
  • Calculated fields: Compute final grades using database functions
  • Grade overrides: Allow manual grade adjustments
Enrollment Pattern

Handle student-course relationships efficiently

  • Junction table: Separate table for student-course relationships
  • Enrollment dates: Track when students join/leave courses
  • Status tracking: Active, completed, dropped, withdrawn
  • Bulk operations: Efficient enrollment management
Performance Optimization Tips
Indexing Strategy
  • Primary keys (auto-indexed)
  • Foreign key columns
  • Frequently filtered columns
  • Composite indexes for complex queries
Query Optimization
  • Use JOINs instead of multiple queries
  • Avoid SELECT * in production
  • Use database transactions for consistency
  • Implement pagination for large datasets
Monitoring
  • Track slow queries
  • Monitor table sizes and growth
  • Set up alerts for performance issues
  • Regular maintenance (VACUUM, ANALYZE)
Implementation Tips for Students
Start Simple

Begin with basic CRUD operations for each entity. Add relationships and complex queries gradually as you understand the data flow.

Test Thoroughly

Create comprehensive tests for your database operations. Test edge cases, constraints, and data integrity.

Plan for Scale

Design your schema to handle growth. Consider how the system will perform with hundreds or thousands of users.
